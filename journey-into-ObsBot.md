# Journey Into ObsBot

**一個 Telegram Bot + Obsidian Vault 知識管理系統的 48 小時開發紀實**

> 基於 claude-mem 持久化記憶系統的 287 筆觀察記錄，橫跨 2026 年 3 月 27 日至 28 日，共 130 個 session。

---

## 1. 專案起源 (Project Genesis)

ObsBot 的故事並非從零開始，而是始於一次「重生」。

2026 年 3 月 27 日下午 4:14，第一筆被記錄的觀察（Obs #1）揭示了一個已經存在的專案——它原本名為 **GetThreads**。一次全面性的審計發現了超過 100 處仍然引用舊名稱的程式碼（橫跨 70+ 個檔案），這標誌著 ObsBot 正式誕生的起點。

ObsBot 要解決的問題很明確：**將散落在各處的網路資訊（GitHub 專案、Reddit 貼文、Twitter 影片、小紅書、微博、抖音等）自動擷取、分類、豐富化，然後存入 Obsidian Vault 作為個人知識庫。** 這不只是一個爬蟲，而是一套完整的知識管理 pipeline：extractor（擷取）→ classifier（分類）→ formatter（格式化）→ saver（存檔），並且透過 Telegram Bot 作為主要互動介面。

初始的架構決策已經相當成熟：
- **不使用任何 LLM SDK**，所有 AI 呼叫走本機 oMLX REST API 或外部 CLI
- **三層模型分級**：flash（4B）、standard（9B）、deep（27B），根據任務複雜度選用
- **Apple Silicon 原生**：whisper.cpp 用 Metal GPU 加速做語音轉文字
- **TypeScript 全棧**，嚴格的 300 行上限與零 `any` 原則

但這個專案有一個重要的歷史包袱——它是從 Windows 環境遷移到 macOS 的。這個遷移的痕跡將在接下來的 48 小時內被逐一清除。

## 2. 架構演進 (Architectural Evolution)

### 第一階段：大遷徙（3/27 下午 4:14 - 6:22）

ObsBot 的第一個重大工程任務不是寫新功能，而是徹底清除舊世界的痕跡。

**GetThreads → ObsBot 品牌遷移**（Obs #1-14）是一場系統性的搜尋與替換行動。從 `package.json` 的 name 欄位、README.md 的品牌標誌、到瀏覽器 session 識別碼（Obs #6：從 `getthreads` 改為 `obsbot`），每一處都必須精確更新。變數名稱的規範化（Obs #12）甚至延伸到了工具腳本中。

緊接著是更深層的 **Windows → macOS 平台遷移**（Obs #19-58）。這個遷移的規模令人驚訝：
- 87 個檔案帶有 Windows CRLF 換行符（Obs #19）
- 31 個檔案帶有 UTF-8 BOM 標記（Obs #48）
- `server.ts` 中的中文註解已經損壞成亂碼（Obs #15）
- 技能文件中充斥著 `netstat`、`taskkill` 等 Windows 專用指令（Obs #26-37）

最終，99 個檔案的換行符被正規化為 LF（Obs #51），31 個 BOM 標記被移除（Obs #52），所有程序管理指令都替換為 Unix 工具（`ps aux`、`pkill`、`lsof`）。這是一場基礎設施級別的遷移，卻在短短兩小時內完成。

### 第二階段：功能爆發（3/27 下午 6:22 - 9:50）

平台遷移完成後，開發進入了高速功能迭代期。

**指令架構審查**（Obs #59-65）首先對現有的 13 個 Claude 技能和 28 個 Telegram Bot 指令進行全面盤點。這次審查不只是文件更新——它發現了 `command-runner.ts` 中的雙分號語法錯誤（Obs #61），並催生了分層幫助系統（Obs #64），將 35 行的完整指令清單精簡為 8 個核心指令的入門版本。

**Discover 內嵌儲存按鈕**（Obs #67-102）是當天最重要的功能特性。它將「發現 GitHub 專案 → 手動複製 URL → 貼到聊天視窗」的繁瑣流程，簡化為一鍵「存入」。這個功能的實現涉及 URL 處理管線的重構（Obs #67：提取為共享服務）、SHA1 截斷的 token 快取機制、以及 Telegram inline keyboard 的整合。

**Admin Dashboard 大改造**（Obs #103-127）將簡陋的設定頁面升級為功能完整的監控面板：暗色主題、即時伺服器狀態、一鍵管理、Vault 統計、日誌串流——這是一次從設定工具到運維儀表板的架構躍遷。

### 第三階段：智慧化與優化（3/28 下午 3:39 - 10:18）

第二天的開發重心轉向了系統的智慧化層面。

**智慧推薦引擎**（Obs #145-159）是最大的新功能：透過分析 Vault 內容建立興趣模型，結合平台特定的探索策略，每天上午 10 點自動推送個人化推薦。這個功能充分利用了現有的排程器和 AI 基礎設施（Obs #146），展現了成熟架構的擴展能力。

**oMLX 模型選擇優化**（Obs #276-287）則是架構層面的精煉。全面審計發現 `selectModelTier()` 無條件將所有 GitHub 內容、連結文章、逐字稿都路由到昂貴的 27B 模型，即使內容很短也是如此。新的長度感知邏輯將 deep 模型使用率從約 40% 降至估計 15-20%（Obs #283），同時為每個呼叫點加上了 `maxTokens` 約束和超時優化（Obs #284）。

## 3. 關鍵突破 (Key Breakthroughs)

### 突破一：Discover 回調管線的完整貫通

從 Obs #67 到 Obs #99，經歷了超過 30 筆觀察和 20 個 session，Discover 內嵌儲存按鈕才真正端對端地運作。這個過程中的每一步都揭示了一個新問題：

1. 回調處理器只顯示「已儲存」但沒有實際執行儲存（Obs #77）
2. 缺少 `processUrlToVault` 的 import（Obs #80）
3. GitHub 擷取器超時（Obs #83-88）
4. 回調管線走了錯誤的程式碼路徑（Obs #89）
5. `chatId` 在回調上下文中缺失（Obs #92）
6. `handleUrl` 收到的是 Markdown 格式連結而非原始 URL（Obs #96）

這不是一個「寫完就能用」的功能——它是一場關於 Telegram Bot API 回調機制、非同步管線架構、以及資料格式一致性的深度學習。最終在 Obs #99 確認完整運作時，這個功能已經經歷了 6 次獨立的 bug 修復。

### 突破二：Twitter 影片 URL 的堆疊溢位修復

當系統收到 `video.twimg.com` 的直接影片 URL 時，整個 bot 崩潰了——Maximum call stack size exceeded。

根本原因的追查（Obs #235）揭示了一個精妙的 bug：web-extractor 的四層 fallback 系統在 Tier 0 成功解析重導向後，將結果存在 `finalUrl` 中，但 Tier 1 和 Tier 2 的瀏覽器 fallback 卻使用了原始的 `url` 參數，導致無限重導向迴圈。

修復方案不是簡單的一行修改，而是建立了三層防禦（Obs #234-237）：
1. HTTP 錯誤現在優雅地降級而非直接拋出例外
2. 二進制內容類型在解析前就被偵測並拒絕
3. URL 預驗證機制在任何處理開始前就攔截媒體檔案連結

最終，這個問題催生了全新的 **Direct Video Extractor**（Obs #247），專門處理原始影片 URL，繞過 HTML 解析層直接進入下載→分析→存檔流程。

### 突破三：分類器重複資料夾的根因分析

Obs #263-272 記錄了一場教科書級的根因分析。使用者發現 Vault 中出現了重複的「RAG & 知識圖譜」資料夾。

調查揭示了一個三層因果鏈：
1. **動態學習系統**（Obs #263）從 Vault 內容中學到了「AI/RAG & 知識圖譜」這個分類
2. **Frontmatter 與資料夾路徑不一致**（Obs #264）：筆記的 YAML frontmatter 保留了 `&` 符號，但資料夾路徑被清理掉了
3. **資料夾路徑清理正則表達式**（Obs #266）會移除 `&` 但保留空格，導致「RAG & 知識圖譜」變成「RAG  知識圖譜」（雙空格）

修復同樣是多層的（Obs #267）：修改清理正則允許 `&`、加上空白正規化、新增靜態分類、並重新分類 13 筆被錯誤放置的筆記（Obs #273）。

## 4. 工作模式 (Work Patterns)

### 密集的 Session 節奏

整個開發過程記錄了 **130 個 session**（S1-S130），涵蓋 **287 筆觀察**。平均每個 session 產生約 2.2 筆觀察，但分佈極不均勻——最活躍的 session 產生了 58 筆觀察。

開發明顯分為兩天：
- **3/27（下午 4:14 - 晚上 9:50）**：約 5.5 小時，專注於遷移、指令審查、Discover 功能、Admin Dashboard
- **3/28（下午 3:39 - 晚上 10:18）**：約 6.5 小時，專注於診斷修復、推薦引擎、影片處理、模型優化

### 「發現-修復-驗證」三拍子

幾乎每個功能和修復都遵循相同的節奏：

1. **Discovery（發現）**：用 agent 探索程式碼結構，識別問題範圍
2. **Implementation（實作）**：修改程式碼、編譯驗證（`npx tsc --noEmit`）
3. **Deployment & Verification（部署驗證）**：重啟 bot、端對端測試、確認生產環境正常

這個模式在 Admin Dashboard 的開發中特別明顯：每個子功能（狀態 API、Vault 統計、日誌串流、快速操作）都經歷了獨立的 deploy-verify 迴圈（Obs #112→113、#114→115、#119→120、#123→125）。

### 功能開發的「洋蔥層」模式

大型功能往往展現出層層展開的特性。以 Discover 為例：

- **第一層**：基本功能實作（Obs #67-72）
- **第二層**：回調機制除錯（Obs #75-99）
- **第三層**：跨平台測試（Obs #101-102）

每一層都假設前一層已經完成，但實際上每一層都暴露了新的假設錯誤。

## 5. 技術債 (Technical Debt)

### 已清償的技術債

**Windows 遺產**是最大的技術債項目。從 CRLF 換行符到 `taskkill` 指令、從 BOM 標記到 Windows 路徑分隔符，這些遺產橫跨了整個程式碼庫。清償時間：約 2 小時（Obs #19-58），但在第二天仍然發現了遺漏（Obs #45：`measure_bodies.mjs` 中的 Windows 路徑）。

**雙分號語法錯誤**（Obs #61）是一個微小但有啟示意義的債務——它說明即使是無害的語法瑕疵，在全面審查中也會被揪出。

**ProcessGuardian 的 logOut 炸彈**（Obs #252-261）是最危險的技術債。409 衝突的恢復邏輯會呼叫 Telegram 的 `logOut` API，導致 bot token 被永久失效 20 分鐘以上。這個債務在 3/28 晚間被引爆，最終需要透過 @BotFather 重新生成 token 才能解決（Obs #259-261）。修復方案（Obs #254）統一使用 `deleteWebhook` 策略取代 `logOut`。

### 識別但尚未完全清償的債務

**分類器的關鍵字匹配**仍然存在 substring 匹配陷阱的風險。雖然 Obs #89 加入了語言感知的最小長度驗證（ASCII 至少 5 字元、CJK 至少 3 字元），但 57% 的 RAG 筆記被錯誤分類（Obs #272）顯示分類器的準確度仍有改善空間。

**瀏覽器程序洩漏**（Obs #160-163）被識別並修復，但這類資源管理問題往往在高負載下才會顯現真正的影響。

## 6. 挑戰與除錯長征 (Challenges and Debugging Sagas)

### 長征一：Discover 回調的七重關卡

這是整個開發過程中最漫長的除錯歷程，從 S26 到 S43，橫跨 18 個 session。

**關卡 1**（S26, Obs #75）：Telegram 回調的 response timing 問題——按鈕按下後沒有視覺回饋。
**關卡 2**（S28, Obs #77）：回調處理器是空殼——只更新按鈕文字為「已儲存」但沒有執行任何動作。
**關卡 3**（S31, Obs #80-81）：修復後仍然靜默失敗，追查到 `processUrlToVault` 的 import 遺漏。
**關卡 4**（S33-S36, Obs #83-89）：GitHub 擷取器超時。`gh` CLI 認證正常、擷取器獨立測試正常——問題出在回調管線走了不同的程式碼路徑（`processUrlToVault` vs `bot.handleUrl`）。
**關卡 5**（S39, Obs #92）：改用 `bot.handleUrl` 後，`chatId` 在回調上下文中缺失。
**關卡 6**（S41, Obs #95-96）：URL 驗證失敗——`handleUrl` 期望原始 URL，但收到的是 Telegram Markdown 格式的連結文字。
**關卡 7**（S42-S43, Obs #97-99）：URL 提取邏輯修正，最終確認端對端運作。

每一關都需要獨立的診斷、假設、修復、部署、驗證循環。這段歷程完美展示了為什麼「寫完就能用」是一個危險的假設。

### 長征二：Twitter 影片的崩潰與重生

從 S98 到 S109，這段歷程從一個堆疊溢位錯誤開始，最終催生了一個全新的 extractor。

**第一幕：崩潰分析**（Obs #234-235）。堆疊溢位的根因是 web-extractor 的 fallback 鏈在 URL 解析與重導向之間產生了不一致，導致無限迴圈。

**第二幕：多層防禦**（Obs #236-237）。三道防線被建立：graceful degradation、binary content detection、URL pre-validation。但這只是防禦性修復——影片 URL 仍然無法被實際處理。

**第三幕：新架構**（Obs #243-250）。發現系統已有完整的影片處理管線（yt-dlp + FFmpeg + Whisper），決定建立專用的 Direct Video Extractor，在 web-extractor 之前攔截影片 URL。

**第四幕：409 災難**（Obs #252-261）。部署新 extractor 時觸發了 Telegram 409 衝突，ProcessGuardian 的 `logOut` 恢復策略反而讓 token 失效。最終需要重新生成 token、重構恢復策略、並重新部署。

這段歷程展示了一個典型的工程現實：修復一個 bug 可能觸發另一個完全不相關的問題。

### 長征三：分類器的幽靈資料夾

Obs #262-273 記錄了一個看似簡單（重複資料夾）但根因複雜的問題。

它涉及三個獨立的子系統：
1. **動態學習器**從 Vault 內容中學習分類
2. **Frontmatter 解析器**讀取筆記的 YAML 元資料
3. **資料夾路徑清理器**將分類名轉為安全的檔案路徑

當「RAG & 知識圖譜」這個包含 `&` 符號的分類名進入這三個子系統時，每個系統對 `&` 的處理方式不同，最終導致了資料不一致。修復需要同時調整三個系統，並手動重新分類 13 筆受影響的筆記。

## 7. 記憶與延續性 (Memory and Continuity)

### 跨 Session 的知識傳遞

在 130 個 session 中，claude-mem 的持久化記憶發揮了關鍵作用。每個 session 結束時自動記錄的觀察，成為下一個 session 的啟動知識。

最明顯的例子是 Discover 回調的除錯過程。S33 發現 GitHub 擷取器超時（Obs #83），S34 確認 `gh` CLI 認證正常（Obs #87），S35 驗證擷取器獨立運作正常（Obs #88），S36 最終定位到管線路徑差異（Obs #89）。每個 session 都能立即接續前一個的發現，而非重新探索。

### 記憶驅動的架構決策

Obs #145-146 展示了記憶如何影響架構決策。在設計推薦引擎時，系統記住了現有的排程器基礎設施和 AI 呼叫模式，直接決定將推薦引擎整合到現有架構中，而非建立獨立系統。

同樣地，Obs #243 記錄了「ObsBot 已經有完整的影片處理管線」這個發現，這個記憶直接影響了 Direct Video Extractor 的設計——它不需要重新實作下載和轉錄邏輯，只需要建立正確的入口點。

### 使用者偏好的記憶

MEMORY.md 中記錄的使用者偏好（繁體中文回覆、UTF-8 編碼、分類器要理解內容而非只抓關鍵字）在整個開發過程中持續影響行為。Obs #89 中分類器的語言感知關鍵字驗證，就是對「分類器必須理解內容重點」這個反饋的直接回應。

### 記憶 ROI 分析

**節省率 98%** 意味著 287 筆觀察中儲存的 5,466,272 tokens 的知識，在後續使用時只需要 124,411 tokens 就能被召回。這相當於每投入 1 token 進行記錄，就能在未來節省約 44 tokens 的重新探索成本。

**顯式召回事件**僅有 2 次被明確記錄在觀察的 narrative 中，但實際的記憶使用遠超這個數字——每個 session 啟動時自動注入的上下文，以及時間線查詢，都是隱式的記憶利用。

## 10. 教訓與反思 (Lessons and Meta-Observations)

### 教訓一：遷移是一場永遠做不完的清掃

從 GetThreads 到 ObsBot 的品牌遷移、從 Windows 到 macOS 的平台遷移，這兩場遷移在「完成」之後仍然不斷冒出遺漏。Obs #45 在「完成」後 1 小時才發現 `measure_bodies.mjs` 中的 Windows 路徑；Obs #48 在已經提交 git 之後才發現 24 個檔案的 CRLF 需要由 `.gitattributes` 自動修正。

**對新開發者的啟示**：永遠不要假設遷移已經「完成」。建立自動化驗證（grep 搜索、CI 檢查）比人工確認更可靠。

### 教訓二：回調機制是 Bug 的溫床

Discover 功能的七重關卡（Obs #75-99）揭示了一個模式：當使用者互動觸發非同步回調時，資料在傳遞過程中每一步都可能出錯。token 過期、chatId 缺失、URL 格式不匹配、管線路徑分歧——這些都不是邏輯錯誤，而是介面合約（interface contract）的隱性假設被違反。

**對新開發者的啟示**：為每個回調建立端對端的整合測試。單元測試無法捕捉管線路徑分歧這類問題。

### 教訓三：防禦性 API 呼叫可能比問題本身更危險

ProcessGuardian 的 `logOut` 恢復策略（Obs #252-261）是一個經典的「cure worse than the disease」案例。為了處理 409 衝突，系統呼叫了 `logOut` API，結果讓 bot token 永久失效 20 分鐘，最終需要人工介入重新生成 token。

**對新開發者的啟示**：在設計自動恢復策略時，永遠評估最壞情況。`deleteWebhook` 是可逆的；`logOut` 不是。選擇可逆的恢復手段。

### 教訓四：分類器需要多維度驗證

分類器的問題不只出現一次：Obs #89 的 substring 匹配（`ads` 匹配到 `attachments`）、Obs #271 的 LightRAG 被誤分類為「其他」、Obs #272 的 57% 筆記放錯資料夾。這些問題的共同點是：分類器在添加新規則時缺乏回歸測試。

**對新開發者的啟示**：每次修改分類器規則後，必須對整個 Vault 執行分類回歸測試。CLAUDE.md 中的這條規則不是建議——它是血的教訓。

### 教訓五：48 小時可以做很多事

在短短 48 小時（實際工作約 12 小時）內，ObsBot 完成了：
- 兩場完整的平台遷移
- 一次全面的指令架構審查
- 三個重大新功能（Discover 內嵌儲存、Admin Dashboard、智慧推薦）
- 一個新的 extractor（Direct Video）
- 一次架構級優化（oMLX 模型選擇）
- 無數 bug 修復和程序管理改進

這個產出量得益於幾個因素：成熟的 pipeline 架構讓新功能可以「插入」而非「重建」；持久化記憶消除了跨 session 的上下文重建成本；嚴格的「診斷先行」原則避免了盲目猜測導致的返工。

### 最終反思

ObsBot 的開發故事不是一個英雄式的從零到一的創造敘事。它是一個更真實的工程故事：繼承遺留系統、清理技術債、在現有架構上疊加新功能、處理生產環境中的意外崩潰、以及在每一次失敗中學習更好的防禦策略。

287 筆觀察記錄了 5,466,272 tokens 的工作量，但最有價值的不是這些 token——而是它們背後記錄的決策脈絡、失敗教訓、和架構直覺。這些記憶讓每一個新的 session 都能站在前一個 session 的肩膀上，而不是從零開始。
