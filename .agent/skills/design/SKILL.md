# /design — 架構確認 + 可行性驗證

整合原 `/preflight` 的 POC 驗證。30 秒確認方向 → 必要時 POC 驗證 → 確認後才寫碼。

## 使用方式

```
/design              # 架構確認（適合明確的功能）
/design --poc        # 架構確認 + POC 驗證（適合不確定可行性的功能）
```

---

## 核心規則

- **絕對不跳過確認**：必須得到用戶明確「確認」才開始寫碼
- **展示，不猜測**：列出受影響的檔案和函數
- **簡短精準**：確認表不超過 10 行
- POC 最大 **30 行**，不做完整功能

---

## Phase 1：一句話描述

> 「在 [哪裡] 加入 [什麼功能]，當 [觸發條件] 時 [行為]」

## Phase 2：架構決策表

```
📐 架構確認

功能：[一句話描述]
類型：[ ] 新獨立指令  [ ] 整合到現有流程  [ ] 修改現有行為
入口點：[哪個檔案的哪個函數]
新增檔案：[列出，或「無」]
修改檔案：[列出具體檔案和函數]
影響範圍：[哪些現有功能可能受影響]
```

## Phase 3：可行性判斷

根據功能特性判斷是否需要 POC：

| 情況 | 是否需要 POC |
|------|------------|
| 純邏輯修改（分類規則、格式化） | ❌ 不需要 |
| 新增外部依賴或 API | ✅ 需要 |
| 不確定執行時間或記憶體 | ✅ 需要 |
| 涉及 Camoufox / yt-dlp 新用法 | ✅ 需要 |

不需要 POC → 跳到 Phase 5。

## Phase 4：POC 驗證（原 /preflight）

### 4a：建立最小 POC

建立 `preflight-test.ts`（≤ 30 行）：

```typescript
// 只測試核心假設
const start = Date.now();
const res = await fetch('https://example.com/api', { signal: AbortSignal.timeout(10_000) });
console.log(`✅ 狀態：${res.status}，時間：${Date.now() - start}ms`);
console.log(`記憶體：${Math.round(process.memoryUsage().heapUsed / 1024 / 1024)}MB`);
```

### 4b：執行 + 評估

```bash
npx ts-node preflight-test.ts
```

| 指標 | 目標 | 實測 | 結果 |
|-----|------|-----|------|
| 執行時間 | < 30s | ? | ✅/❌ |
| 記憶體 | 合理 | ? | ✅/❌ |
| Windows 相容 | 是 | ? | ✅/❌ |

- **全部通過** → 繼續
- **任一失敗** → 停止，提出替代方案

### 4c：清除 POC

```bash
rm preflight-test.ts
```

## Phase 5：等待確認

> 「以上方向正確嗎？確認後開始實作。」

只有收到明確確認才開始寫碼。

## 常見陷阱（歷史教訓）

| 陷阱 | 正確做法 |
|------|---------|
| 「加入評論功能」→ 做成 `/comments` 指令 | 先問：整合到 URL 處理，還是獨立指令？ |
| 「加入翻譯」→ 做成同步處理 | 先問：同步還是非同步？ |
| 「改善搜尋」→ 新增搜尋引擎 | 先問：改善 DDG 還是加新引擎？ |
| 「用 AI 分類」→ 直接用 LLM | 先 POC 驗證硬體能否跑 |

## 已知硬體限制

- 不使用本地大型語言模型
- HTTP 超時 30s、yt-dlp 120s
- Telegraf handler 上限 90 秒
